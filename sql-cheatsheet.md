# 🗄️ Database Concepts — Deep Reference Guide

> A practical reference covering architectural, structural, and behavioral database concepts.  
> No syntax. No `CREATE TABLE`. Just concepts that make you think like a database engineer.

---

## 📐 Data Modeling & Structure

| Concept                               | Explanation                                                                                                                                                                                                                                                  | Use Cases                                                                                                                                | Scenario                                                                                                                                                                                                            | Edge Cases for Learning                                                                                                                                         |
|---------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Normalization**                     | The process of organizing a database to reduce redundancy and improve data integrity by dividing large tables into smaller, related ones following a set of rules called Normal Forms (1NF, 2NF, 3NF, BCNF).                                                 | Transactional systems (banking, e-commerce), any system where data consistency matters more than read speed.                             | A `users` table stores city and country. Instead of repeating "Accra, Ghana" for 10,000 users, you create a `locations` table and reference it. Update "Ghana" once — it reflects everywhere.                       | Over-normalizing a reporting database causes painful multi-join queries. Ask: is this an OLTP system (normalize) or OLAP system (maybe don't)?                  |
| **Denormalization**                   | Intentionally introducing redundancy into a schema by merging tables or duplicating data to improve read performance at the cost of write complexity and storage.                                                                                            | Read-heavy systems, reporting dashboards, data warehouses, anywhere query speed matters more than storage efficiency.                    | An e-commerce app joins `orders`, `users`, and `products` on every page load. You denormalize by storing `user_name` and `product_name` directly in the `orders` table, eliminating joins.                          | Denormalization creates update anomalies — if a product name changes, you must update it in dozens of rows. Always weigh read gain vs. write pain.              |
| **Entity-Relationship (ER) Modeling** | A visual and conceptual method of designing a database by identifying entities (things), their attributes (properties), and the relationships between them before writing any schema.                                                                        | Early-stage system design, communicating DB structure to non-technical stakeholders, onboarding new engineers.                           | Designing a school system: `Student`, `Course`, and `Teacher` are entities. A student *enrolls in* many courses, and a teacher *teaches* many courses — these become relationships with cardinality (1:N, M:N).     | Many-to-many relationships always require a junction/bridge table. Missing this in ER modeling causes schema mistakes downstream.                               |
| **Composite Key**                     | A primary key made up of two or more columns that together uniquely identify a row, used when no single column alone can guarantee uniqueness.                                                                                                               | Junction/bridge tables, enrollment records, order line items, any M:N relationship.                                                      | In a `student_courses` table, neither `student_id` nor `course_id` alone is unique — a student takes many courses and a course has many students. Together, `(student_id, course_id)` is the composite primary key. | Adding a third column to a composite key "just in case" is a smell. Also, foreign keys referencing composite keys must carry all columns — easy to get wrong.   |
| **Surrogate Key vs Natural Key**      | A **surrogate key** is a system-generated, meaningless identifier (e.g., auto-increment ID, UUID). A **natural key** is a real-world identifier that exists in the domain (e.g., email, national ID, ISBN).                                                  | Surrogate keys: most transactional tables. Natural keys: lookup tables, data imports, systems that integrate with external data sources. | A `users` table uses `id SERIAL` (surrogate) as PK, but also has a `UNIQUE` constraint on `email` (natural key). The surrogate key is stable even if the email changes.                                             | Natural keys seem convenient but break when the real world changes (email changes, government re-issues IDs). Surrogate keys add a join but are more resilient. |
| **Star Schema vs Snowflake Schema**   | Two approaches to organizing data in a data warehouse. **Star schema**: a central fact table linked directly to flat dimension tables (denormalized). **Snowflake**: dimension tables are further normalized into sub-dimensions, forming a snowflake shape. | Star: BI tools, dashboards, fast aggregation queries. Snowflake: when dimension data is large, complex, or shares sub-dimensions.        | Star: `sales_fact` links to `date_dim`, `product_dim`, `store_dim`. Snowflake: `product_dim` further links to `category_dim` and `brand_dim`.                                                                       | Star is faster to query but uses more storage. Snowflake saves storage but adds joins. Most modern BI tools prefer star schema for speed.                       |

## ⚡ Performance & Scaling

| Concept                                | Explanation                                                                                                                                                                                                                             | Use Cases                                                                                                                                    | Scenario                                                                                                                                                                                                                                                      | Edge Cases for Learning                                                                                                                                                                                                                                                                  |
|----------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Sharding (Horizontal Partitioning)** | Splitting a large database into smaller, independent pieces called shards, each stored on a separate server. Each shard holds a subset of the data (e.g., by user ID range or geography).                                               | Massive-scale applications (social networks, global SaaS), when a single DB server can no longer handle the data volume or write throughput. | A messaging app has 500M users. Sharding by `user_id % 10` creates 10 shards — shard 0 holds users 0,10,20..., shard 1 holds users 1,11,21..., etc. Each shard is a separate DB server.                                                                       | Cross-shard queries (e.g., "find all users who messaged user X across all shards") are extremely expensive. Re-sharding (changing the shard key) is painful and often requires downtime.                                                                                                 |
| **Vertical vs Horizontal Scaling**     | **Vertical scaling** (scale up): adding more CPU, RAM, or storage to a single server. **Horizontal scaling** (scale out): adding more servers to distribute the load.                                                                   | Vertical: small-to-medium workloads, easier to implement. Horizontal: large-scale systems, cloud-native architectures.                       | A startup's DB gets slow — they upgrade from 16GB to 64GB RAM (vertical). Years later, one server isn't enough, so they add read replicas and shard (horizontal).                                                                                             | Vertical scaling has a hard ceiling (you can't add infinite RAM to one machine). Horizontal scaling introduces distributed systems complexity — coordination, consistency, and network failures.                                                                                         |
| **Partitioning**                       | Dividing a single large table into smaller physical segments (partitions) within the same database server, based on a partition key such as a date range, list of values, or hash. Unlike sharding, partitions live on the same server. | Time-series data, logs, archive tables — any table that grows indefinitely in one direction.                                                 | An `orders` table is partitioned by year: `orders_2023`, `orders_2024`, `orders_2025`. A query for "orders in 2024" only scans the 2024 partition, not the entire table.                                                                                      | Partition pruning only works when the query filters on the partition key. Querying without the partition key does a full scan across all partitions — sometimes worse than no partitioning.                                                                                              |
| **Indexing Strategy**                  | The deliberate decision of which columns to index, what type of index to use (B-tree, hash, composite, partial, covering), and how indexes affect both read and write performance.                                                      | Any table with frequent lookups, filtering, sorting, or joining on specific columns.                                                         | A `products` table is queried by `category` AND `price`. A composite index on `(category, price)` satisfies both filters in one index scan. A covering index also includes `name` so the query never hits the main table.                                     | Indexes speed up reads but slow down writes (every INSERT/UPDATE must update all relevant indexes). Over-indexing a write-heavy table kills performance. The order of columns in a composite index matters — `(category, price)` ≠ `(price, category)`.                                  |
| **Query Execution Plan**               | The step-by-step strategy the database query optimizer chooses to execute a SQL query — which indexes to use, in what order to join tables, and how to filter rows. Exposed via `EXPLAIN` / `EXPLAIN ANALYZE`.                          | Performance debugging, query optimization, understanding why a query is slow.                                                                | You run a query expecting it to use an index but it's slow. The execution plan shows a `Seq Scan` (full table scan) instead of `Index Scan` — revealing the optimizer chose not to use the index, perhaps because the table is small or statistics are stale. | The optimizer can make bad choices if table statistics are outdated. Running `ANALYZE` refreshes them. Sometimes manually hinting the optimizer (forcing an index) is necessary but should be a last resort.                                                                             |
| **Caching Layer**                      | A fast, in-memory data store (like Redis or Memcached) placed in front of the database to serve frequently read data without hitting the DB, dramatically reducing latency and load.                                                    | Read-heavy applications, session storage, leaderboards, rate limiting, API response caching.                                                 | An e-commerce homepage shows the top 10 products. Instead of querying the DB on every page load, the result is cached in Redis with a 5-minute TTL. The DB is only hit every 5 minutes instead of thousands of times.                                         | Cache invalidation is the hardest problem — when does stale cache data become a bug? Strategies: TTL expiry, write-through (update cache on write), cache-aside (app checks cache first). Cache stampede: TTL expires for a hot key and thousands of requests hit the DB simultaneously. |

## 🔒 Reliability & Consistency

| Concept | Explanation | Use Cases | Scenario | Edge Cases for Learning |
|---|---|---|---|---|
| **ACID Properties** | A set of four guarantees that ensure database transactions are processed reliably: **Atomicity** (all or nothing), **Consistency** (data remains valid), **Isolation** (transactions don't interfere), **Durability** (committed data survives crashes). | Financial transactions, booking systems, inventory management — anywhere partial failure is catastrophic. | Transferring $500 from Account A to Account B: debit A and credit B must both succeed or both fail (Atomicity). If the server crashes after debiting A but before crediting B, the DB rolls back the debit (Durability + Atomicity). | Full ACID compliance is expensive. Isolation levels (Read Committed, Repeatable Read, Serializable) trade consistency for performance. Serializable is fully ACID but slowest. |
| **BASE Properties** | An alternative to ACID used in NoSQL/distributed systems: **Basically Available** (system stays up), **Soft state** (data may change over time without input), **Eventual consistency** (data will *eventually* become consistent across nodes). | Distributed NoSQL systems (Cassandra, DynamoDB), social media feeds, systems prioritizing availability over immediate consistency. | A user posts on a social network. Their follower in another country may see the post 2 seconds later than a local follower — the system is eventually consistent, not immediately consistent. | BASE doesn't mean "no consistency." It means relaxed consistency with guarantees it'll converge. The business must decide: is it okay for a user to briefly see stale data? For a bank — no. For a like count — yes. |
| **Transactions & Isolation Levels** | Isolation levels define how visible the intermediate state of one transaction is to others. Levels (weakest to strongest): **Read Uncommitted → Read Committed → Repeatable Read → Serializable**. Each prevents different anomalies: dirty reads, non-repeatable reads, phantom reads. | Any multi-user system where concurrent transactions touch the same data. | Two users book the last seat on a flight simultaneously. At `Read Committed`, both might read "1 seat available" and proceed. At `Serializable`, the second transaction waits or fails, preventing double-booking. | Higher isolation = fewer anomalies but more locking and lower throughput. Most production databases default to `Read Committed`. Knowing *which anomaly* your use case can tolerate is key to choosing the right level. |
| **Deadlocks** | A situation where two or more transactions are each waiting for the other to release a lock, creating a cycle where neither can proceed. The database detects this and kills one transaction (the victim). | Any system with concurrent write transactions on multiple rows/tables. | Transaction A locks Row 1, then tries to lock Row 2. Transaction B locks Row 2, then tries to lock Row 1. Both wait forever — deadlock. The DB kills one transaction and the application must retry it. | Deadlocks are often solved by enforcing a consistent lock ordering (always lock resources in the same order). Keeping transactions short and touching fewer rows reduces deadlock probability. |
| **CAP Theorem** | In a distributed system, you can only guarantee **two of three** properties simultaneously: **Consistency** (every read gets the latest write), **Availability** (every request gets a response), **Partition Tolerance** (system works despite network splits). Since network partitions are unavoidable, the real choice is **CP vs AP**. | Distributed database design — choosing between systems like Zookeeper (CP) and Cassandra (AP). | Network split separates two DB nodes. A CP system (like HBase) refuses reads/writes on the minority side to avoid inconsistency. An AP system (like Cassandra) keeps serving requests but might return stale data. | CAP is often misunderstood as a permanent choice. Modern systems (like Google Spanner) blur the line by minimizing partition probability. CAP also ignores latency — consistency sometimes just means slow, not unavailable. |

## 🏗️ Architecture & Patterns

| Concept                           | Explanation                                                                                                                                                                                                                                                                                                                   | Use Cases                                                                                                      | Scenario                                                                                                                                                                                                                                                                     | Edge Cases for Learning                                                                                                                                                                                                                                                 |
|-----------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Replication (Primary-Replica)** | Copying data from one database server (primary/master) to one or more other servers (replicas/slaves) in real time or near-real time. Writes go to primary; replicas stay in sync.                                                                                                                                            | High availability, disaster recovery, read scaling, geographic distribution.                                   | An e-commerce app has one primary DB handling all writes and three read replicas serving product search queries. If the primary fails, one replica is promoted to primary.                                                                                                   | Replication lag is the silent killer — replicas may be milliseconds to seconds behind the primary. A user writes data then immediately reads from a replica and sees old data. "Read your own writes" consistency is a common requirement that replication can violate. |
| **Read Replicas**                 | A specific use of replication where replica servers are dedicated solely to serving read queries, offloading read traffic from the primary and improving overall throughput.                                                                                                                                                  | Analytics queries, reporting, read-heavy applications, search features.                                        | A social media app's primary DB handles all writes (posts, likes). Three read replicas handle all `SELECT` queries for timelines and profiles. The primary is barely touched by reads, improving write performance.                                                          | Directing the right queries to replicas requires application-level logic or a proxy. Heavy analytics queries on replicas can still slow them down and cause replication lag, indirectly affecting the primary's sync.                                                   |
| **Connection Pooling**            | Maintaining a pool of pre-established database connections that are reused across requests, rather than opening and closing a new connection for every query. Managed by a pooler (e.g., PgBouncer, HikariCP).                                                                                                                | Any web application, especially high-traffic APIs where connection overhead is significant.                    | A web app gets 10,000 requests/second. Opening a new DB connection per request would overwhelm PostgreSQL (which handles ~500 concurrent connections). A connection pool of 100 connections serves all 10,000 requests by queuing and reusing them.                          | Pool exhaustion: all connections are in use and new requests queue up or time out. Setting the pool size too high is also dangerous — each connection consumes DB server RAM. The right pool size depends on DB server specs and query duration.                        |
| **Database Federation**           | Splitting a database by function — different databases handle different domains of the application (e.g., one DB for users, one for products, one for payments), each independently scaled and managed.                                                                                                                       | Large microservices architectures, systems where different parts have very different scaling needs.            | An enterprise app has `UserDB` (PostgreSQL), `ProductDB` (MongoDB), and `PaymentsDB` (PostgreSQL with strict ACID). Each service owns its database — no shared DB.                                                                                                           | Joins across federated databases are impossible at the DB level — they must happen in application code, which is expensive. Distributed transactions (payments touching both UserDB and PaymentsDB) are very hard to coordinate correctly.                              |
| **OLTP vs OLAP**                  | **OLTP** (Online Transaction Processing): optimized for fast, frequent, small read/write transactions. **OLAP** (Online Analytical Processing): optimized for complex queries over large historical datasets for analysis. These are fundamentally different workloads requiring different schema designs and infrastructure. | OLTP: e-commerce checkouts, banking, booking systems. OLAP: business intelligence, reporting, data warehouses. | An airline's booking system (OLTP) processes thousands of seat reservations per second with simple queries. The analytics team runs OLAP queries on 5 years of booking data to find seasonal trends — these queries scan millions of rows and would destroy the OLTP system. | Running analytical queries on an OLTP database is a classic scaling mistake. The solution is an ETL pipeline that moves OLTP data into an OLAP warehouse (e.g., BigQuery, Redshift, Snowflake) where heavy queries don't affect production.                             |

## 🧠 Advanced Concepts

| Concept                       | Explanation                                                                                                                                                                                                                                              | Use Cases                                                                                                                        | Scenario                                                                                                                                                                                                                                           | Edge Cases for Learning                                                                                                                                                                                                                                                             |
|-------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Eventual Consistency**      | A consistency model in distributed systems where, given no new updates, all replicas of the data will *eventually* converge to the same value — but at any point in time, different nodes may have different values.                                     | NoSQL databases (DynamoDB, Cassandra), distributed caches, collaborative apps, social feeds.                                     | User A updates their profile picture. Node 1 (US server) applies the change immediately. Node 2 (EU server) gets the update 800ms later due to replication lag. Any EU user who requests the profile in that window sees the old picture.          | "Eventually" has no time bound by definition — it could be milliseconds or minutes depending on network conditions. Systems must handle the case where a user reads stale data right after writing, often using techniques like sticky sessions or read-your-own-writes guarantees. |
| **Write-Ahead Logging (WAL)** | A durability and recovery mechanism where every change to the database is first written to an append-only log file before being applied to the actual data pages on disk. On crash, the DB replays the log to recover.                                   | Every serious relational database uses WAL internally (PostgreSQL, MySQL InnoDB, SQLite). Also used for replication.             | PostgreSQL writes "set account balance to $500" to the WAL log first. If the server crashes mid-write before the data page is flushed to disk, PostgreSQL replays the WAL on restart and reaches the correct state — no data loss.                 | WAL is also the mechanism that powers logical replication and change data capture (CDC). WAL files can grow very large if not archived or cleaned. In PostgreSQL, `wal_level` settings affect how much information is logged — important for replication setup.                     |
| **Materialized Views**        | A database object that stores the *result* of a query physically on disk, like a cached table. Unlike regular views (which re-run the query every time), materialized views must be explicitly or periodically refreshed to reflect new underlying data. | Pre-aggregating expensive analytics queries, dashboards, reporting, any scenario where the same complex query is run frequently. | A dashboard shows total revenue by country for the last 30 days — a query joining 5 tables and scanning millions of rows. Instead of running it on every dashboard load, a materialized view stores the result and is refreshed every hour.        | A stale materialized view can serve dangerously outdated data without any warning. Concurrent refresh (`REFRESH MATERIALIZED VIEW CONCURRENTLY`) avoids locking but requires a unique index. Triggering refresh on every write defeats the purpose.                                 |
| **Cursor**                    | A database mechanism that allows row-by-row iteration over a query result set, rather than fetching all rows at once into memory. The DB holds the result set server-side while the client fetches it in chunks.                                         | Processing large result sets that would exceed memory if fully loaded, ETL pipelines, batch processing, streaming exports.       | An ETL job needs to export 10 million rows from a `transactions` table. Loading all 10M rows into application memory would crash the server. A cursor fetches 1,000 rows at a time, processes them, and advances — stable memory usage throughout. | Cursors hold a transaction open for their entire lifetime in some databases — a long-running cursor can block vacuuming in PostgreSQL or hold locks that slow other queries. Always close cursors when done and keep their scope short.                                             |
| **N+1 Query Problem**         | A performance anti-pattern where fetching a list of N items results in 1 query to get the list, then N additional queries to fetch related data for each item — totalling N+1 database round trips instead of 1 or 2.                                    | Any ORM-backed application (Rails, Django, Hibernate, Prisma) that uses lazy loading of associations.                            | Fetching 100 blog posts and then, for each post, running a separate query to get the author's name = 101 queries. The fix is eager loading — a single JOIN or a second query that fetches all 100 authors at once.                                 | The N+1 problem is almost invisible in development (small datasets, fast local DB) but catastrophic in production (N=10,000 means 10,001 queries per request). Always monitor query counts per request, not just total query time.                                                  |
