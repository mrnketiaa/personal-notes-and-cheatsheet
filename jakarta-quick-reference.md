# 📖 Jakarta Persistence 3.1 — Plain English Lookup Guide

> A layman-friendly reference table for every major concept in the Jakarta Persistence 3.1 specification.
> Think of this as a cheat sheet — read this first, then go deep in the official docs.

## Chapter 1 — Introduction

|-------|--------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1.0   | **Introduction**         | Jakarta Persistence (formerly JPA) is the standard Java way to talk to a relational database using Java objects instead of raw SQL. You write Java classes, and JPA handles translating them to database tables. | Any Java/Jakarta EE application that needs to store and retrieve data from a relational database (MySQL, PostgreSQL, Oracle, etc.). | JPA is a *specification*, not an implementation. ## Chapter 2 — Entities

| Index | Concept                                          | Explanation (Plain English)                                                                                                                                                                                                                                                | Use Cases                                                                                                                                                                                                | Edge Cases for Learning                                                                                                                                                                                                                                |
|-------|--------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 2.0   | **Entities**                                     | An entity is a plain Java class that represents a row in a database table. You mark it with `@Entity` and JPA knows to treat it as something that should be saved to/loaded from the database.                                                                             | Every object you want to store in the database — User, Product, Order, Invoice, etc.                                                                                                                     | Not every class should be an entity. Value objects (things without independent identity, like an Address embedded in a User) should be `@Embeddable`, not `@Entity`.                                                                                   |
| 2.1   | **The Entity Class**                             | Rules for what a valid entity class looks like: it must have a no-argument constructor, must not be `final`, must not have `final` methods or `final` fields that JPA needs to access. It also needs a primary key field.                                                  | Defining any new table-backed object in your app.                                                                                                                                                        | A common beginner mistake is making the class or its getter methods `final`. JPA providers use bytecode tricks (proxying) to intercept calls — `final` blocks this and causes confusing runtime errors.                                                |
| 2.2   | **Persistent Fields and Properties**             | JPA can read/write data from an entity in two ways: directly from the **fields** (instance variables) or through **getter/setter methods** (properties). You pick one approach per class.                                                                                  | Field access is more common and cleaner. Property access is used when you need validation logic in your getters.                                                                                         | Mixing field and property access in the same class without explicitly declaring it causes unpredictable behavior. Stick to one.                                                                                                                        |
| 2.3   | **Access Type**                                  | The choice between field access vs. property access. JPA needs to know which style you're using so it knows where to look when reading/writing your entity's data.                                                                                                         | Mostly a one-time decision per project or class.                                                                                                                                                         | If you annotate `@Id` on a field → field access. If you annotate `@Id` on a getter → property access. Where you put `@Id` is what decides the access type, not just any random annotation.                                                             |
| 2.4   | **Primary Keys and Entity Identity**             | Every entity must have a primary key — the unique identifier that distinguishes one row from another. This is marked with `@Id`. The primary key is also how JPA finds, updates, and deletes specific records.                                                             | Every entity, no exceptions.                                                                                                                                                                             | Two entity instances with the same primary key value *are* considered the same entity by JPA, even if they're different Java objects in memory. This is what "entity identity" means.                                                                  |
| 2.5   | **Basic Type**                                   | A "basic type" is a simple, single-column field — a `String`, `int`, `boolean`, `Date`, `UUID`, etc. These map directly to a single database column with no special handling needed.                                                                                       | Any simple data field: names, prices, dates, flags, counts.                                                                                                                                              | Not all Java types are automatically basic types. Custom classes need a `@Converter` to become basic types. Also, `enum` fields need `@Enumerated` to tell JPA how to store them.                                                                      |
| 2.6   | **Embeddable Classes**                           | An embeddable is a Java class whose fields get stored in the *same table* as the parent entity — not in their own table. Think of it as grouping related columns under one Java object.                                                                                    | Grouping related fields like `Address` (street, city, zip) inside a `User` entity — stored in the users table, not a separate addresses table.                                                           | Two entities cannot share the same embeddable instance — they get their own copy. Also, an embeddable can't have its own `@Id`. If you need identity, make it an `@Entity` instead.                                                                    |
| 2.7   | **Collections of Embeddables and Basic Types**   | You can have a list/set of embeddable objects or basic values (like a list of strings) on an entity. JPA stores these in a separate *collection table* linked back to the parent.                                                                                          | Storing a user's multiple phone numbers, a product's list of tags, a person's list of addresses.                                                                                                         | The collection table has no identity of its own — its rows are completely tied to the owning entity. If you delete the entity, all collection rows are deleted. If you need the items to be shared or independently managed, use a separate `@Entity`. |
| 2.8   | **Map Collections**                              | An entity field can be a `Map` — a key-value pair collection. Both the keys and values can be basic types, embeddables, or even other entities. JPA stores the map in a separate table.                                                                                    | Storing settings or properties as key-value pairs, like `Map<String, String> preferences`.                                                                                                               | Map keys must be unique per entity instance. If you use an entity as the key, JPA creates a join column for the key. Getting the mapping wrong here leads to confusing schema generation.                                                              |
| 2.9   | **Mapping Defaults for Non-Relationship Fields** | If you don't add any annotation to a field, JPA uses sensible defaults: the column name matches the field name, the field is persisted, etc. You only need annotations when you want to *change* the defaults.                                                             | Quick prototyping where the default column names and types are acceptable.                                                                                                                               | "Convention over configuration" sounds great until the database already has a schema with different column names. Always be explicit with `@Column(name=...)` when working with an existing DB.                                                        |
| 2.10  | **Entity Relationships**                         | Entities can be related to each other just like tables are. JPA supports: **OneToOne**, **OneToMany**, **ManyToOne**, **ManyToMany**. These tell JPA how to link entities using foreign keys.                                                                              | A `User` has many `Orders` (OneToMany). An `Order` belongs to one `User` (ManyToOne). A `Student` takes many `Courses` and a `Course` has many `Students` (ManyToMany).                                  | Every relationship has an *owning side* and an *inverse side*. The owning side controls the foreign key. If you update the inverse side only, JPA won't save the relationship — a very common and frustrating bug.                                     |
| 2.11  | **Relationship Mapping Defaults**                | When you declare a relationship, JPA uses default column and table names based on your class and field names. For example, a `ManyToOne` to `User` will look for a `user_id` column by default.                                                                            | Rapid development when default naming is acceptable.                                                                                                                                                     | Default join column names are generated by JPA but may not match your existing DB schema. Always verify what column name JPA is expecting vs. what's in your database using the schema DDL output.                                                     |
| 2.12  | **Inheritance**                                  | Entity classes can extend other entity classes, just like normal Java inheritance. JPA has to figure out how to store the parent and child class data in the database, and it offers three strategies for this.                                                            | Modeling things like `Animal → Dog`, `Animal → Cat`, or `Payment → CreditCardPayment`, `Payment → BankTransferPayment`.                                                                                  | When you query the parent class, you get back instances of all child types (polymorphic query). This is powerful but can cause full-table scans if not designed carefully.                                                                             |
| 2.13  | **Inheritance Mapping Strategies**               | Three ways JPA can store an inheritance hierarchy in the database: **Single Table** (all classes in one table with a discriminator column), **Joined** (parent and child each have their own table), **Table per Concrete Class** (each child has its own complete table). | Single Table: fast queries, simple, but wastes nullable columns. Joined: clean schema, but requires joins. Table per Concrete Class: no joins for concrete queries, but polymorphic queries are painful. | Single Table is the default and works best for most cases. Joined looks clean but can produce horrible multi-join queries for deep hierarchies. Table per Concrete Class is rarely a good choice.                                                      |
| 2.14  | **Naming of Database Objects**                   | Rules for how JPA names tables, columns, sequences, etc. by default — and how you can override them. The default is usually the class name for tables and field name for columns.                                                                                          | Any time you need your Java names to differ from your DB names (e.g., `firstName` field → `first_name` column).                                                                                          | If you're using a reserved SQL keyword as a field name (like `order`, `value`, `select`), JPA may fail silently or generate broken SQL. Always check generated DDL against the actual DB.                                                              |

## Chapter 3 — Entity Operations

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 3.0 | **Entity Operations** | This chapter covers everything you *do* with entities — saving, loading, updating, deleting, and how JPA manages their lifecycle. | The day-to-day work of persisting data in any JPA application. | Most bugs in JPA apps come from misunderstanding entity lifecycle states. Understanding this chapter well saves hours of debugging. |
| 3.1 | **EntityManager** | The `EntityManager` is your main tool for working with JPA. It's the object you use to save, find, update, and delete entities. Think of it as your "database session" object. | Every database operation in JPA goes through the EntityManager. | The `EntityManager` is **not thread-safe**. Never share one instance between threads. In web apps, each request should get its own EntityManager (or use container-managed injection). |
| 3.2 | **Entity Instance Lifecycle** | Every entity goes through states: **New** (just created, not tracked), **Managed** (tracked by JPA, changes auto-saved), **Detached** (was tracked, no longer is), **Removed** (marked for deletion). | Understanding why a change you made to an entity didn't get saved — it was probably Detached. | The most common bug: you load an entity, pass it to another method or layer, the EntityManager closes, and then you change the entity. JPA doesn't see that change because the entity is now Detached. You need to re-attach it with `merge()`. |
| 3.3 | **Persistence Context Lifetime** | The persistence context is the "bubble" in which JPA tracks your entities. It has a lifetime — either tied to a transaction (short) or extended (long-lived). When the context closes, all entities become Detached. | Understanding when entities are tracked vs. not tracked across different parts of your application. | An extended persistence context lives longer than a transaction, which is powerful but dangerous — it can hold stale data and cause memory issues if you're not careful. |
| 3.4 | **Locking and Concurrency** | When two users try to update the same record at the same time, you need a strategy to prevent data corruption. JPA offers **Optimistic Locking** (assume no conflict, check at save time) and **Pessimistic Locking** (lock the row immediately, block others). | Any feature where multiple users could edit the same data — inventory systems, booking systems, collaborative editing. | Optimistic locking uses a `@Version` field. If two users load the same entity and both try to save, the second one gets an `OptimisticLockException`. You must handle this exception and retry or show an error. |
| 3.5 | **Entity Listeners and Callback Methods** | You can hook into lifecycle events — JPA will call your method automatically before/after save, update, delete, or load. These are `@PrePersist`, `@PostPersist`, `@PreUpdate`, `@PostUpdate`, `@PreRemove`, `@PostRemove`, `@PostLoad`. | Auto-setting `createdAt` / `updatedAt` timestamps, logging, auditing, triggering events. | You cannot call `EntityManager` methods inside a lifecycle callback — it causes errors. Also, exceptions thrown in callbacks will roll back the transaction. |
| 3.6 | **Bean Validation** | JPA integrates with Jakarta Bean Validation so that your entity field constraints (`@NotNull`, `@Size`, `@Min`) are automatically checked before JPA tries to save to the database. | Ensuring data quality at the persistence layer — a last line of defense before bad data hits the DB. | Bean Validation runs by default on `PrePersist` and `PreUpdate`. Disabling it (`validation-mode=none`) is possible but risky — you may end up with corrupt data if you're relying on it as a safety net. |
| 3.7 | **Entity Graphs** | An Entity Graph lets you tell JPA exactly which related entities and fields to load eagerly for a specific query — without changing the class-level fetch settings. It's a query-time override. | Avoiding the N+1 problem by loading all necessary associations in one go, but only when you need them. | Entity graphs are hints, not guarantees — the provider may ignore them. Also, over-fetching with entity graphs (loading everything eagerly everywhere) defeats the purpose and creates performance problems. |
| 3.8 | **Type Conversion of Basic Attributes** | Sometimes you want to store a Java type in a different DB format — for example, store a `boolean` as `'Y'`/`'N'` in the DB, or an `enum` as an int. `@Converter` lets you write the conversion logic. | Legacy databases with unusual formats, encrypting sensitive fields before storing, mapping non-standard types. | A converter that throws an exception on bad input can corrupt entire transactions. Always handle null values explicitly and test converters with edge-case inputs. |
| 3.9 | **Caching** | JPA has a second-level cache (L2 cache) that stores entity data in memory across multiple EntityManager instances, reducing database hits for frequently-read data. The first-level cache is per-EntityManager session. | Read-heavy apps where the same entities are loaded repeatedly — product catalogs, configuration data, reference data. | The L2 cache can serve stale data if the database is modified outside of JPA (e.g., by a script or another app). Always disable the L2 cache for data that changes frequently or from multiple sources. |
| 3.10 | **Query APIs** | JPA provides multiple ways to write queries: **JPQL** (JPA's own SQL-like language that works on entities), **Criteria API** (programmatic, type-safe queries in Java), and **Native SQL** (raw SQL for when you need DB-specific features). | JPQL: most common queries. Criteria API: dynamic queries built at runtime. Native SQL: complex queries, stored procedures, DB-specific functions. | JPQL operates on **entity names and field names**, not table names and column names. Writing `SELECT * FROM users` in JPQL will fail — it should be `SELECT u FROM User u`. This trips up everyone new to JPA. |
| 3.11 | **Summary of Exceptions** | A reference list of all JPA exceptions and what they mean — `EntityNotFoundException`, `OptimisticLockException`, `TransactionRequiredException`, `NonUniqueResultException`, etc. | Debugging production errors and writing proper exception handling. | `NonUniqueResultException` is a sign your query logic is wrong — you expected one result but got many. `LazyInitializationException` (Hibernate-specific) means you tried to access a lazy-loaded relationship after the session closed. |

## Chapter 4 — Query Language (JPQL)

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 4.0 | **Query Language** | JPQL (Jakarta Persistence Query Language) is JPA's own query language. It looks like SQL but operates on your Java entity classes and their fields — not directly on database tables and columns. | Whenever you need to find, filter, sort, or aggregate entity data beyond simple `find(id)` lookups. | JPQL is translated into SQL by the JPA provider. The generated SQL may not be optimal. Always check generated SQL in development using `show_sql=true` or equivalent logging. |
| 4.1 | **Overview** | JPQL queries are strings that reference entity names, their fields, and relationships. They're portable across databases because JPA translates them to the right SQL dialect for each DB. | Writing queries that work on MySQL, PostgreSQL, and Oracle without changing any code. | JPQL does not support all SQL features. For DB-specific features (window functions, CTEs, full-text search), you need Native SQL queries. |
| 4.2 | **Statement Types** | JPQL has three types: **SELECT** (fetch data), **UPDATE** (bulk update), and **DELETE** (bulk delete). SELECT is the most common. Bulk UPDATE and DELETE bypass entity lifecycle events. | SELECT: loading entities. Bulk UPDATE/DELETE: mass data operations like "deactivate all expired accounts". | Bulk UPDATE and DELETE do **not** trigger `@PreUpdate`, `@PostUpdate` callbacks, and they bypass the first-level cache. Your in-memory entities won't reflect bulk changes until you refresh them. |
| 4.3 | **Abstract Schema Types and Query Domains** | In JPQL, the "schema" is your entity model. When you write `FROM User u`, JPQL's schema type for `User` covers all its mapped fields and relationships — this is what you can navigate in the query. | Understanding what you can and can't reference in a JPQL query. | You can only query entities that are part of the persistence unit. Unmapped classes, views, and arbitrary tables are not queryable through JPQL unless mapped as entities. |
| 4.4 | **The FROM Clause and Navigational Declarations** | The `FROM` clause declares which entities you're querying and gives them aliases. You can also traverse relationships using dot notation — called "path expressions" — to navigate from one entity to its related entities. | Any JPQL query that involves multiple entities or filtering/selecting related data. | Navigating a path that involves a collection (like `u.orders.items`) in a `WHERE` clause without a proper `JOIN` causes an error. You must use an explicit `JOIN` to traverse collection relationships. |
| 4.5 | **WHERE Clause** | Filters which entity instances are returned, just like SQL's WHERE. You compare fields, check for nulls, use IN lists, LIKE patterns, and subqueries. | Any time you don't want all rows — filtering by status, date range, user, category, etc. | JPQL's WHERE uses Java-style `null` comparison with `IS NULL` / `IS NOT NULL`, same as SQL. But `= null` does NOT work in JPQL — it never matches anything, just like in SQL. |
| 4.6 | **Conditional Expressions** | All the building blocks of a WHERE clause: comparisons (`=`, `<`, `>`), logical operators (`AND`, `OR`, `NOT`), BETWEEN, IN, LIKE, IS NULL, EXISTS, subqueries, and arithmetic. | Writing any complex filtering logic in JPQL queries. | The `LIKE` wildcard in JPQL uses `%` (any characters) and `_` (single character), same as SQL. To search for a literal `%`, you must escape it — but the escape character itself needs to be declared with `ESCAPE`. |
| 4.7 | **GROUP BY and HAVING** | Aggregates results by one or more fields. `GROUP BY` groups rows, `HAVING` filters groups (like WHERE but applied after grouping). Used with aggregate functions like COUNT, SUM, AVG, MAX, MIN. | Reporting queries: "total sales by category", "number of orders per customer", "average order value per month". | Every field in the SELECT clause must either be in GROUP BY or be inside an aggregate function. Forgetting this causes errors. HAVING applies to groups, not individual rows — don't confuse it with WHERE. |
| 4.8 | **SELECT Clause** | Defines what to return. You can return whole entities, individual fields, aggregates, or even construct new objects using a constructor expression (`NEW com.example.dto.MyDto(u.name, count(o))`). | Returning only the fields you need (instead of full entities) for performance, building DTOs from queries. | Returning full entities (`SELECT u`) is fine for most cases but fetches all columns. For large entities with many columns or blob fields, selecting specific fields into a DTO is much more efficient. |
| 4.9 | **ORDER BY Clause** | Sorts query results by one or more fields, ascending (`ASC`) or descending (`DESC`). | Any time you want results in a specific order — newest first, alphabetical, highest price first. | In JPQL, you can only ORDER BY fields that are in the SELECT clause when using aggregate queries. Also, the database's default sort for NULLs (NULLS FIRST vs NULLS LAST) varies — JPQL doesn't standardize this. |
| 4.10 | **Bulk Update and Delete** | Lets you update or delete many records in one JPQL statement — without loading each entity into memory one by one. Directly generates a single SQL UPDATE or DELETE. | Mass operations: expire all sessions older than 30 days, deactivate all unverified accounts, delete all soft-deleted records. | These bypass the first-level cache and lifecycle callbacks. After a bulk operation, manually clear the EntityManager (`em.clear()`) so it doesn't serve stale cached versions of the updated/deleted entities. |
| 4.11 | **Null Values** | How JPQL handles null — comparisons with null, nulls in collections, null propagation in arithmetic. JPQL follows SQL's three-valued logic: `TRUE`, `FALSE`, `UNKNOWN` (when null is involved). | Any query where fields might be null and you need correct filtering behavior. | `NULL = NULL` is `UNKNOWN` (not TRUE) in JPQL, same as SQL. Use `IS NULL` to check for nulls. Arithmetic involving null always produces null. These silent null issues cause data loss if you're not careful. |
| 4.12 | **Equality and Comparison Semantics** | Rules for how JPA compares values — entity identity is compared by primary key, enum values by their type, strings by value. | Understanding why two entity instances that look the same might not be "equal" in a JPQL IN list. | Two entity objects in Java can have the same data but be different Java objects. In JPQL, entity equality is always by primary key — so the query is correct, but your Java-side `equals()` method might not reflect the same logic if not overridden properly. |

## Chapter 5 — Metamodel API

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 5.0 | **Metamodel API** | The Metamodel is a programmatic, type-safe representation of your entity model. Instead of referring to fields as strings (like `"firstName"`), you use generated Java classes that represent each attribute. Used together with the Criteria API. | Building type-safe dynamic queries that won't compile if you mistype a field name — unlike string-based JPQL. | The Metamodel classes are usually auto-generated by your JPA provider at build time. If your IDE or build tool doesn't generate them, you'll have to fall back to string-based attribute references. |
| 5.1 | **Metamodel API Interfaces** | The set of interfaces that make up the Metamodel: `EntityType`, `ManagedType`, `Attribute`, `SingularAttribute`, `PluralAttribute`, `CollectionAttribute`, etc. Each describes a different aspect of your entity model in a type-safe way. | Writing reusable, generic repository or query-builder code that works across multiple entity types. | The Metamodel reflects the JPA mapping, not the Java class structure. Transient fields and non-mapped fields do not appear in the Metamodel. Trying to access them will throw an exception at runtime. |

## Chapter 6 — Criteria API

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 6.0 | **Criteria API** | A programmatic, Java-code-based way to build JPQL queries. Instead of writing a query string, you build the query by calling Java methods. This makes queries type-safe and dynamically composable. | Building queries where the conditions aren't known until runtime — search filters, dynamic sorting, optional WHERE conditions. | Criteria API queries are verbose and harder to read than JPQL strings. Only use it when you genuinely need dynamic queries. For static queries, JPQL strings (or named queries) are much cleaner. |
| 6.1 | **Overview** | The Criteria API mirrors all of JPQL's capabilities — SELECT, WHERE, JOIN, GROUP BY, ORDER BY — but expressed as Java method calls on `CriteriaBuilder`, `CriteriaQuery`, and `Root` objects. | Anywhere you'd use JPQL but need to build the query conditionally based on user input. | A common trap: building a Criteria query inside a loop or adding the same predicate twice leads to double-conditions in the WHERE clause. Always check generated SQL when debugging unexpected Criteria query results. |
| 6.2 | **Metamodel** | The Criteria API works best when combined with the static Metamodel — auto-generated classes that represent your entity attributes. This gives you compile-time safety so a mistyped field name fails at compile time, not at runtime. | Enterprise applications where refactoring (renaming entity fields) should cause compile errors, not silent runtime failures. | Without the static Metamodel, you can use strings for attribute names (`cb.equal(root.get("firstName"), "John")`), but then you lose type-safety and get runtime errors if the field name changes. |
| 6.3 | **Criteria API Interfaces** | The toolkit: `CriteriaBuilder` (creates expressions, predicates, and queries), `CriteriaQuery` (the query itself), `Root` (your starting entity), `Join` (linked entities), `Predicate` (conditions), `Expression` (values and calculations). | Understanding which object to use for which part of a Criteria query. | `CriteriaBuilder` is obtained from the `EntityManager` and is thread-safe. `CriteriaQuery` and `Root` are query-specific and should not be reused across queries. |
| 6.4 | **Criteria Query API Usage** | How to put the pieces together: get a `CriteriaBuilder`, create a `CriteriaQuery`, define a `Root`, add `WHERE` predicates, set `ORDER BY`, execute via `TypedQuery`. | The standard template for any Criteria-based query. | Forgetting to call `cq.select(root)` when you want entities back is a common mistake — without it, the query may not know what to return. |
| 6.5 | **Constructing Criteria Queries** | Step-by-step how to build each part of a Criteria query: roots, joins, filters, grouping, ordering, parameter binding, subqueries. | Building complex, multi-condition search screens or filter pages. | Dynamic predicates are usually collected in a `List<Predicate>` and combined with `cb.and(predicates.toArray(...))`. Adding null predicates to this list causes NullPointerExceptions — always null-check conditions before adding them. |

## Chapter 7 — Entity Managers and Persistence Contexts

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 7.0 | **Entity Managers and Persistence Contexts** | Deep dive into how `EntityManager` instances are created, scoped, and managed — either by the container (Jakarta EE) or by your application code manually. | Understanding the lifecycle of your JPA session and why entities sometimes behave unexpectedly. | The single biggest source of JPA bugs is confusion about which persistence context is active, and whether the entity you're holding is managed or detached. |
| 7.1 | **Persistence Contexts** | A persistence context is a short-lived "working area" where JPA tracks entity objects. Any entity loaded within this context is "managed" — JPA watches it for changes and syncs them to the DB at flush time. | Every JPA operation happens within a persistence context, whether you think about it or not. | Two different persistence contexts are completely isolated from each other. An entity loaded in context A is invisible to context B. This is why passing entities across service boundaries (or HTTP requests) leads to detached entity problems. |
| 7.2 | **Obtaining an EntityManager** | Two ways to get one: **Container-managed** (inject it with `@PersistenceContext` — the server handles its lifecycle) or **Application-managed** (you create and close it yourself via `EntityManagerFactory`). | Container-managed: standard Jakarta EE apps, Spring apps. Application-managed: Java SE, testing, background jobs, custom scopes. | Never close a container-managed EntityManager manually — the container owns it. Closing an application-managed EntityManager without calling `close()` leaks connections. |
| 7.3 | **Obtaining an Entity Manager Factory** | The `EntityManagerFactory` is the expensive, long-lived object that represents a database connection pool + JPA setup. You create it once at startup and use it to create individual `EntityManager` instances. | Application startup configuration, testing setup, Java SE apps without a container. | Creating an `EntityManagerFactory` is slow (seconds). Creating an `EntityManager` from it is fast (milliseconds). Never create a new `EntityManagerFactory` per request — it will kill your app's performance. |
| 7.4 | **EntityManagerFactory Interface** | The API for the factory object — `createEntityManager()`, `close()`, `getCache()`, `getProperties()`, etc. | Programmatic control over the persistence unit at runtime. | Always call `factory.close()` when the application shuts down to release the DB connection pool. Forgetting this leaks connections and causes "too many connections" errors in production. |
| 7.5 | **Controlling Transactions** | How to begin, commit, and roll back database transactions using JPA. In Jakarta EE, transactions are usually managed by the container (JTA). In Java SE or Spring, you manage them yourself via `EntityTransaction`. | Any write operation — saves, updates, deletes. Reading without a transaction is possible but risky (lazy loading issues). | Always commit or roll back a transaction in a `finally` block or use try-with-resources patterns. A transaction left open holds DB locks and eventually causes timeouts or deadlocks. |
| 7.6 | **Container-managed Persistence Contexts** | When running inside a Jakarta EE container (WildFly, Payara, etc.), the container creates and destroys persistence contexts for you, usually tied to a transaction. You just inject `@PersistenceContext` and use it. | Standard enterprise Java apps, EJBs, CDI beans in Jakarta EE environments. | By default, a container-managed context is transaction-scoped — it dies when the transaction ends. Any entity you loaded becomes Detached after the transaction. Extended persistence contexts survive beyond a single transaction but require careful management. |
| 7.7 | **Application-managed Persistence Contexts** | You manually create, use, and close the `EntityManager`. Used in environments without a container — Java SE apps, unit tests, standalone batch jobs. | Testing, Java SE applications, any code running outside of a Jakarta EE / Spring container. | You are fully responsible for transactions and cleanup. Forgetting to close the EntityManager in tests leads to connection pool exhaustion. Always use try-finally or JUnit lifecycle hooks to ensure cleanup. |

## Chapter 8 — Entity Packaging

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 8.0 | **Entity Packaging** | How you package your JPA entities and configuration into a deployable unit — called a Persistence Unit — and how you tell JPA where to find your entities and how to connect to the database. | Setting up any JPA project from scratch. | Packaging mistakes (wrong JAR structure, missing `persistence.xml` location) cause JPA to silently find no entities. Always verify entity discovery works in early development. |
| 8.1 | **Persistence Unit** | A named, self-contained JPA configuration that defines: which entities belong to it, the database connection, caching settings, and validation mode. Defined in `persistence.xml`. | Every JPA project has at least one persistence unit. Multi-tenant apps or apps connecting to multiple databases may have several. | Two persistence units in the same app can share entities, but those entities can get confusing to manage. Prefer one persistence unit per data source. |
| 8.2 | **Persistence Unit Packaging** | The rules for where `persistence.xml` must live, what JARs are scanned for entities, and how to control which classes are included or excluded. | Building projects with a clean separation of entity JARs and application JARs, OSGi deployments, modular apps. | If `exclude-unlisted-classes` is set to `true`, JPA only includes classes explicitly listed in `persistence.xml`. Forgetting to add a new entity there means it gets silently ignored — no error, it just doesn't save to the DB. |

## Chapter 9 — Container and Provider Contracts

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 9.0 | **Container and Provider Contracts** | The rules that JPA providers (Hibernate, EclipseLink) and Jakarta EE containers must follow to work together correctly — how they bootstrap, integrate with transactions, and generate schemas. | Understanding why JPA behaves differently in different environments (WildFly vs Tomcat vs Spring Boot). | Hibernate running standalone (Java SE) behaves differently from Hibernate inside WildFly (container-managed). Understanding this contract explains many environment-specific bugs. |
| 9.1 | **Jakarta EE Deployment** | How JPA providers are discovered and initialized when deployed inside a Jakarta EE server. The server handles starting JPA, connecting to the datasource, and managing transactions. | Deploying WAR or EAR applications to application servers. | In Jakarta EE, the datasource is configured on the server (not in `persistence.xml`) and referenced by JNDI name. Misconfiguring the JNDI name leads to datasource-not-found errors at startup. |
| 9.2 | **Bootstrapping in Java SE** | How to start JPA manually in a Java SE environment using `Persistence.createEntityManagerFactory("unit-name")`. | 9.4 | **Schema Generation** | JPA can automatically create, update, or validate your database schema based on your entity mappings. Controlled by `jakarta.persistence.schema-generation.database.action`. | Development and testing: `create-drop` creates the schema fresh and deletes it after. ## Chapter 10 — Metadata Annotations

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 10.0 | **Metadata Annotations** | The full catalog of JPA annotations — the `@Something` tags you put on your classes and fields to configure mappings, queries, and behavior without writing XML. | The day-to-day toolbox for any JPA developer. | Annotations are processed at startup. Invalid annotation configurations usually throw exceptions at startup, not at query time — which is good for catching mistakes early. |
| 10.1 | **`@Entity`** | Marks a Java class as a JPA entity — tells JPA "this class represents a table in the database." | Every class that maps to a DB table. | The class must have a no-arg constructor (can be protected or public). Without it, JPA can't instantiate objects when loading from DB. It doesn't need to be in your application code — JPA uses it internally. |
| 10.2 | **Callback Annotations** | The lifecycle hook annotations: `@PrePersist`, `@PostPersist`, `@PreUpdate`, `@PostUpdate`, `@PreRemove`, `@PostRemove`, `@PostLoad`. Placed on methods that should be called at the corresponding lifecycle event. | Auto-stamping timestamps, logging, initializing transient fields after load, enforcing business rules. | `@PostLoad` is called after an entity is loaded from the DB — useful for setting up transient calculated fields. But don't do expensive computations here — it runs for *every* entity loaded, which adds up fast. |
| 10.3 | **EntityGraph Annotations** | `@NamedEntityGraph` lets you define a named, reusable entity graph directly on an entity class, which you can then reference by name in queries to control what gets fetched. | Defining standard "fetch profiles" for an entity — a "summary" graph (just basic fields) and a "detail" graph (with all relationships). | Defining too many named graphs makes entities hard to reason about. If a graph is only used in one place, consider using a programmatic entity graph at the query site instead. |
| 10.4 | **Annotations for Queries** | `@NamedQuery` (JPQL), `@NamedNativeQuery` (raw SQL), `@NamedStoredProcedureQuery` — pre-defined, named queries that are compiled at startup and can be referenced by name throughout the app. | Centralizing frequently-used queries in one place, improving performance (parsed once at startup). | Named queries are validated at startup — a typo in JPQL causes the app to fail to start, which is actually a feature. Dynamic string-built queries fail only at runtime when that code path is hit. |
| 10.5 | **References to EntityManager and EntityManagerFactory** | `@PersistenceContext` injects an `EntityManager`. `@PersistenceUnit` injects an `EntityManagerFactory`. These are the standard annotations for getting JPA objects in a container environment. | Any Jakarta EE or CDI-managed bean that needs to interact with the database. | `@PersistenceContext` injects a **proxy**, not the actual EntityManager. The proxy routes calls to the correct context for the current transaction/thread. Never store it in a static field or pass it to non-managed objects. |
| 10.6 | **Annotations for Type Converter Classes** | `@Converter` marks a class as a JPA type converter, and `@Convert` applies it to a specific field. The converter class implements `AttributeConverter<JavaType, DBType>`. | Encrypting fields, storing enums as specific strings, mapping `Money` objects to decimal columns, boolean to Y/N. | A converter that throws exceptions on unexpected DB values (like encountering a NULL in a non-nullable mapped type) will crash every query that touches that column. Always handle null defensively. |

## Chapter 11 — Object/Relational Mapping Metadata

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 11.0 | **O/R Mapping Metadata** | The complete set of annotations that control exactly how Java fields/classes map to DB tables/columns — column names, types, join strategies, keys, indexes, constraints. | Fine-tuning your schema to match existing databases, optimizing for performance, meeting DBAs' naming standards. | The more annotations you add, the more tightly coupled your Java model is to a specific schema. If the schema needs to change, you'll have to update annotations throughout your codebase. |
| 11.1 | **Annotations for O/R Mapping** | The full catalog: `@Table`, `@Column`, `@JoinColumn`, `@JoinTable`, `@OneToMany`, `@ManyToOne`, `@OneToOne`, `@ManyToMany`, `@Id`, `@GeneratedValue`, `@Embeddable`, `@Embedded`, `@Version`, `@Transient`, `@Lob`, `@Enumerated`, `@Temporal`, and many more. | Every entity class and its fields. | `@Transient` tells JPA to ignore a field completely — it won't be stored or loaded. Don't confuse it with Java's `transient` keyword (which is for Java serialization, not JPA). You can have both if needed. |
| 11.2 | **O/R Metadata Used in Schema Generation** | A subset of the mapping annotations that directly affect the DDL JPA generates — table names, column constraints, nullable flags, unique constraints, indexes, foreign key names. | Controlling the exact schema JPA creates when using `schema-generation`. | Generated schema is a starting point, not a production migration tool. Always review generated DDL before running it in production. JPA schema generation doesn't handle data migrations — use Flyway or Liquibase for that. |

## Chapter 12 — XML Object/Relational Mapping Descriptor

| Index | Concept | Explanation (Plain English) | Use Cases | Edge Cases for Learning |
|---|---|---|---|---|
| 12.0 | **XML O/R Mapping Descriptor** | Instead of (or in addition to) annotations on your Java classes, you can define all JPA mappings in an XML file called `orm.xml`. XML mappings can override or supplement annotation-based mappings. | Keeping mapping metadata separate from source code, overriding 3rd-party library mappings you can't annotate, multi-environment configs (different column names per environment). | XML and annotation mappings can conflict. When both exist, XML overrides annotations by default. This can cause confusing behavior if you don't know an `orm.xml` file is present in the classpath. |
| 12.1 | **Use of the XML Descriptor** | You place `orm.xml` in `META-INF/` and JPA picks it up automatically. It uses the same concepts as annotations — entities, columns, relationships — just in XML syntax. | Legacy projects that predated annotation-based mapping, projects where source code cannot be modified. | Forgetting to update the `orm.xml` after refactoring entity class names or field names causes silent mapping failures or startup errors. Annotations are harder to "forget" because they're right next to the code. |
| 12.2 | **XML Overriding Rules** | Precise rules for what XML can override and in what order — persistence-unit defaults → entity-level mappings → field-level mappings. Later rules win. | Multi-environment deployments where dev/test/prod have different schema conventions. | XML overriding is powerful but creates hidden complexity. A developer looking only at annotations won't see the full picture. Always document when XML overrides are in use. |
